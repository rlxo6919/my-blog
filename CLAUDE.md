# 블로그 포스트 작성 가이드

## Frontmatter
- title, date, tags, excerpt 필수
- date 형식: "YYYY-MM-DD"
- tags: 글을 분류하는 **주제 태그** 배열
- excerpt: 검색/공유 시 노출될 한 줄 요약

## 태그 작성 규칙
- `category`가 이미 `study` / `troubleshooting`를 나누므로, `tags`는 더 좁은 **세부 주제**만 담당
- 태그는 기본적으로 `1~3개`만 사용. 정말 핵심인 주제만 남기고, 억지 확장은 금지
- 태그는 "이 글을 어떤 주제로 다시 찾을 것인가?" 기준으로 붙임. "이 글을 어떤 기술로 작성했는가?" 기준으로 붙이지 않음
- `Kotlin`, `Spring Boot`, `JPA`, `MySQL`, `Redis` 같은 **기술 스택/제품명 태그는 기본적으로 사용하지 않음**. 제목과 본문에서 이미 충분히 드러나기 때문
- `데이터베이스`, `성능 최적화`, `채팅`처럼 너무 넓거나 문맥 의존적인 태그는 지양. 태그 페이지에서 묶였을 때 바로 주제가 보여야 함
- 태그 순서는 중요도 순서. 카드/상세 페이지에서 그대로 노출되므로, 가장 대표 주제를 앞에 둠
- 가능하면 기존 태그를 재사용하고, 새 태그는 아래 조건을 만족할 때만 추가:
  - 제목, excerpt, 본문 핵심 섹션에서 반복적으로 다루는 중심 주제일 것
  - 이후 다른 글과 묶일 가능성이 있거나, 단독으로도 검색 의도가 강한 주제일 것
- 단발성 태그는 최대한 줄이되, `정규화`, `커넥션 풀`처럼 **독립 주제로서 검색 의도가 강한 경우**는 예외적으로 허용

### 현재 공통 태그 집합
- `쿼리 최적화`: 조회 비용, 실행 계획, 배치 조회, 응답 경량화, 조회 구조 개선이 핵심인 글
- `N+1`: 반복 조회, 루프 안 쿼리, 건별 조회 제거가 핵심인 글
- `캐시`: 캐시 전략, 키 설계, TTL, 무효화, 캐시 적중/미스가 핵심인 글
- `인덱스`: 인덱스 구조, 인덱스 설계, 인덱스 활용 실패 원인이 핵심인 글
- `실행 계획`: `EXPLAIN`, 옵티마이저 판단, 실행 계획 해석이 핵심인 글
- `동시성 제어`: 락, MVCC, 격리 수준처럼 동시 접근 제어가 핵심인 글
- `트랜잭션`: 트랜잭션 경계, 격리 수준, `FOR UPDATE` 판단 기준이 핵심인 글
- `락`: 잠금 종류, 잠금 범위, 비관적/낙관적 락이 핵심인 글
- `정규화`: 정규형, 함수 종속, 이상 현상이 핵심인 글
- `커넥션 풀`: 풀 크기, 고갈, HikariCP 동작이 핵심인 글

### 태그 선택 예시
- N+1을 배치 조회로 줄이고, 캐시 키도 함께 손봤다면: `["N+1", "캐시"]`
- 실행 계획과 옵티마이저 판단을 읽는 글이라면: `["실행 계획", "쿼리 최적화"]`
- 격리 수준, MVCC, 락 경합처럼 읽기/쓰기 충돌 제어가 핵심이라면: `["동시성 제어"]`를 기본으로 두고, 필요하면 `트랜잭션`, `락`을 보조로 추가
- 특정 모듈명이나 도메인명은 태그보다 제목/본문에서 설명하고, 태그에는 넣지 않음

## 본문 h1 금지
- frontmatter의 title이 페이지 제목으로 렌더링되므로 본문에 `#` 사용 금지
- 본문은 `##`부터 시작

## 글 구조

### category: troubleshooting (실무 경험 기반 개선기)
- 도입부: 독자가 공감할 에러/증상 목록으로 시작 → 글의 범위를 한 문장으로 요약. 배경 설명은 최소화
- 본문: Phase별로 "문제 → 시도/해결" 흐름 반복. 각 Phase는 `## Phase N. 제목` 형식으로, 하위 단계는 `###`으로
- 코드: 변경 전/후를 보여주되, 최종 완성 코드는 마지막 Phase에 한 번만 (중복 금지)
- 마무리: 개선 결과 요약 + 교훈 (번호 목록, **굵은 글씨 규칙** + ` — ` 구분 + 부연 설명, 경어체 유지)
- 대표 리듬: "이상 증상 나열 → 원인 추적 → 구조 변경 → 결과 비교 → 재사용 가능한 교훈 정리"
- 각 Phase는 가능하면 `### 문제:`와 `### 해결:`의 짝으로 끊고, 해결 직후에 왜 이 방식이 유효한지 한 문단으로 설명
- 문제 설명은 에러 메시지 자체보다 "왜 현재 구조가 병목/오동작을 만드는가"를 드러내는 데 집중
- 해결책은 단순 치환보다 구조 변화가 보이게 작성 (`개별 조회 → 배치 조회`, `Service 침투 → API 경계로 이동`, `개별 캐시 → Context/배치 전략`)
- `Before`/`After`, 간단한 흐름도, 개선 전후 표를 적극 사용해 차이를 즉시 보이게 함
- 개선 결과에는 가능하면 쿼리 수, I/O 수, 캐시 적중 여부, 응답 시간 추정치 등 관측 가능한 변화 포함
- 교훈은 "이번 사례에서 끝나는 회고"보다 다른 코드에도 적용 가능한 점검 질문이나 설계 원칙으로 추상화

### category: study (개념 정리/이론 학습)
- 도입부: "왜 이걸 알아야 하는가?"로 시작 → 독자가 겪을 수 있는 문제 상황을 짧게 제시하고 글의 범위를 요약
- 본문: Phase별로 개념을 단계적으로 확장. 각 Phase는 `## Phase N. 제목` 형식으로, 하위 단계는 `###`으로
- 코드/예제: 개념을 설명하는 예제 테이블, SQL, 다이어그램 등 활용. 단계별로 점진적으로 복잡도를 높임
- 마무리: 전체 개념을 비교하는 요약 테이블 + 번호 목록으로 핵심 포인트 정리 (경어체 유지)
- 대표 리듬: "문제 상황 제시 → 개념 정의 → 짧은 예시 → 왜 그런지 설명 → 필요하면 `> **참고:**`로 예외/주의점 보충"

## 톤
- 한국어, 경어체 (~습니다)
- `###` 소제목에 구어체 설명 권장 (예: "### 문제: Redis에 뭐가 들어있는지 모르겠다")
- 기술 용어/코드는 backtick으로 감싸기
- 핵심 개념은 **굵은 글씨**로 강조
- 문장은 짧고 직선적으로 씀. 한 문단에서 한 주장만 전달하고, 바로 아래 예시나 비유로 받쳐줌
- 도입부는 추상 설명보다 "이런 상황에서 문제가 생깁니다" 식의 실무/사용자 시나리오를 먼저 배치
- 설명은 교과서식 정의 나열보다 "왜 중요한가 → 어떻게 동작하는가 → 그래서 무엇을 주의해야 하는가" 순서를 선호
- tone은 단정적이되 과장하지 않음. 핵심 원리는 분명하게 말하고, 엔진/버전/구현체에 따라 달라지는 예외만 좁게 한정
- 비교 항목이 있으면 표보다 먼저 한두 문장으로 큰 차이를 요약하고, 표는 정리용으로 사용
- `study` 글의 소제목은 독자가 바로 이해할 수 있는 질문형/설명형을 선호 (`왜`, `무엇`, `어떻게`, `차이`, `문제`)
- `troubleshooting` 글에서는 "제가", "우리는"보다 시스템/구조를 주어로 두고 서술. 회고문보다 기술 개선 기록에 가깝게 유지
- 개선 서사는 감정 묘사보다 진단 근거와 구조적 이유를 우선. "느렸다"보다 "리뷰 20건에 쿼리 40번"처럼 관측 가능한 문장 선호
- 해결 근거는 기능 설명보다 선택 이유를 붙여서 작성. "적용했다"에서 멈추지 말고 "왜 이 방식이 이전 문제를 없애는지"까지 적기

## 내용 원칙
- 서비스 내부 정보(구체적 캐시 수, 도메인 클래스명 등) 일반화
- 패키지명은 가상이지만 그럴듯하게 (예: `com.mycompany.product`)
- 기술적 정확성 우선 — API 기본 동작, Phase 간 인과관계가 모순 없어야 함
- DBMS, 프레임워크, 라이브러리의 동작 설명은 가능하면 **공식 문서**를 기준으로 작성. 구현체/버전 의존성이 있으면 초반에 기준 버전과 범위를 함께 명시
- DBMS, 프레임워크, 라이브러리별 동작 차이가 있으면 하나로 뭉뚱그려 일반화하지 말고 범위를 명시
- `반드시`, `항상`, `유일한 정답`, `시작이자 끝` 같은 단정 표현은 실제로 보편 타당한 경우에만 사용. 실무 팁은 가능하면 `많은 경우`, `도움이 될 수 있습니다`, `검토할 가치가 큽니다`처럼 표현
- 특정 설정이나 패턴은 "이 사례에서는"이 맞다면 그렇게 한정해서 서술
- `> **참고:**` blockquote로 보충 설명 제공 가능
- 다음 글 예고, 커밋 수 같은 내부 작업 이력 불필요
- 회사명, 실제 서비스명, 기밀 코드, 내부 API 경로 등 회사 식별 가능한 정보 절대 금지. 도메인 클래스명, 패키지명, 엔드포인트 등은 반드시 가명 처리
- 예외 사항은 본문 흐름을 끊지 말고 `> **참고:**`로 분리. 본문은 기본 원리 설명에 집중
- DB, 인덱스, 트랜잭션처럼 오해하기 쉬운 주제는 정확성을 우선하되, 방어적인 표현을 과하게 반복하지 않음
- "정확한데 읽기 어려운 문장"보다 "핵심은 단정적으로, 예외는 짧게 한정한 문장"을 우선
- 같은 단락 안에서 `보통`, `경우에 따라`, `안전합니다` 같은 완충 표현을 여러 번 반복하지 않음
- 하나의 글에서 설명 기준 엔진이나 구현체를 정했으면 초반에 명시하고, 이후에는 그 기준을 유지. 다른 엔진을 함께 다루면 차이점을 별도 섹션/표로 분리
- 요약 섹션은 본문 반복이 아니라 "무엇을 구분해야 하는지"를 다시 잡아주는 문장으로 마무리

## Markdown 렌더링 규칙
- 표는 코드 블록으로 감싸지 말고 순수 Markdown table로 작성
- 표 앞뒤 설명 문장은 코드 펜스 밖에 둔다
- SQL, Kotlin, JSON 등 실제 코드만 fenced code block 사용
- 식별자 성격의 용어(`Service`, `Mapper`, `innerJoin`, `leftJoin` 등)는 본문과 소제목 모두 backtick으로 표기
- SQL 문법 조각, 함수/메서드명, 애노테이션, 에러명, 잠금 모드, 알고리즘/패턴명처럼 **영문 기술 용어를 "이름"으로 다룰 때**도 backtick으로 표기
- 예: `FOR UPDATE`, `GET_LOCK()`, `@Version`, `lock wait timeout`, `deadlock`, `tie-breaker`, `keyset pagination`, `advisory lock`, `Redlock`
- 같은 글 안에서 같은 용어는 처음 등장만이 아니라 **반복 등장 시에도 표기를 일관되게 유지**
- ASCII 다이어그램, 트랜잭션 타임라인, 인덱스 구조도처럼 텍스트 기반 예시는 적극 활용 가능

## 시리즈 문체 기준
- 제목은 `완전 정복`, `기초`, `핵심`처럼 학습 글임이 드러나는 표현을 사용하되 과장된 마케팅 문구는 피함
- 첫 `##`는 거의 항상 `왜 필요한가요?`, `왜 알아야 하나요?`처럼 독자의 필요를 먼저 묻는 형태를 선호
- 도입부 bullet은 보통 3개 전후로 유지하고, 실제로 생길 수 있는 문제만 씀
- 각 Phase는 이전 Phase의 이해를 전제로 자연스럽게 확장되어야 함. 같은 난이도의 개념을 병렬 나열하기보다 계단식으로 전개
- 예시 뒤 설명 문단은 "무슨 일이 일어났는지"를 그대로 한번 풀어쓴 뒤, 마지막 문장에서 용어를 붙이는 패턴을 선호
- 비유는 짧고 1회성으로 사용. 비유 자체를 길게 끌고 가지 않음
- 리스트는 길게 늘이지 말고, 중요 포인트 3~6개 수준으로 끊어 씀

### troubleshooting 문체 기준
- 첫 `##`는 `이런 에러를 만나셨나요?`, `이런 증상을 겪고 계신가요?`처럼 독자가 겪는 현상에서 시작
- 도입부 bullet은 "문제의 종류"가 아니라 "관측된 증상"으로 작성 (`쿼리 40번`, `적중률 0%`, `누락`, `중복`)
- 각 Phase 제목은 "무엇을 고쳤는지"가 드러나야 함 (`N+1 제거`, `캐시 키 수정`, `레이어 책임 정리`)
- `### 문제:` 절에서는 현재 구조와 부작용을 짧게 보여주고, `### 해결:` 절에서는 변경된 패턴을 이름 붙여 제시 (`배치 조회 + Map 매핑`, `Context Resolver`, `Assembler 분리`)
- 코드 예시는 가능하면 `Before`/`After` 대조로 배치하고, 코드가 길면 바로 아래에 핵심 변경 포인트를 짧게 정리
- 실무 글의 `참고` 블록은 프레임워크 함정, 선택 이유, 적용 범위 제한을 설명하는 용도로 사용
- 마무리 표는 단순 나열보다 "문제 축 / 개선 전 / 개선 후 / 가능하면 수치" 형식을 선호

## 메타데이터 점검
- `excerpt`는 첫 화면에서 바로 읽히는 한 문장으로 작성하고, "이 글이 도움이 됩니다" 같은 약한 문구보다 글의 주제와 범위를 직접 설명
- `tags`는 발행 전에 기존 공통 태그 집합으로 먼저 분류해 보고, 정말 필요할 때만 새 태그를 추가
- `tags`는 기술 스택 나열이 아니라 탐색용 주제 분류인지 다시 확인
- 태그가 4개 이상이거나, 너무 넓은 태그 하나에 기대고 있으면 다시 줄이기

## 발행 전 체크
- DBMS, 프레임워크, 라이브러리 동작을 설명한 부분은 가능하면 공식 문서 기준으로 다시 확인
- 가능하면 `npm run build`로 Markdown 파싱, frontmatter, 정적 페이지 생성까지 확인
- 표, 코드 블록, 인용문이 실제 렌더링에서 의도대로 보이는지 우선 점검
